ZigHouse principles

From SQL to SIMD

ZigHouse is an experiment in building a small, native analytical engine in Zig. The idea is simple: store data by column, turn SQL into a compact execution plan, and use memory mapping, vectorized scans, and compile-time specialization to keep the CPU busy.

1. What Kind of Database Is This?

A row store is like a ledger: each row keeps all fields for one event together. A column store splits the ledger into vertical strips: one array for user_id, one for event_date, one for metric_value, and so on. Analytical queries usually touch only a few columns, so columnar execution reads less data.

Columnar

Values of the same type sit next to each other, which is ideal for sequential scans and SIMD reductions.

Native

The engine works close to the operating system: memory maps, typed slices, explicit allocation.

Analytical

The first-class operations are counts, sums, averages, minimums, maximums, filters, and aggregations.

2. The Minimal End-to-End Path

A useful generic engine should not recognize only one hard-coded benchmark query. It should parse a SQL shape, bind names to columns, and choose a numerical kernel. 

SELECT AVG(value_column)
FROM events
WHERE predicate_column <> 0;
SQLThe user writes a query.
ParseThe engine recognizes projection, table, and filter.
PlanThe query becomes a small aggregate plan.
BindColumn names become typed arrays.
ExecuteA SIMD kernel scans and returns rows.

3. Architecture in Plain Terms

Query Layer

  • Parse SQL into a compact plan.
  • Bind column names to actual column data.
  • Select the best computation strategy for each expression.

Storage Layer

  • Keep hot numeric columns in fixed-width binary form.
  • Map column files into memory and treat them as typed arrays.
  • Avoid constructing row objects during scans.

Execution Layer

  • COUNT(*) can often use metadata directly.
  • SUM, AVG, MIN, and MAX scan only the needed columns.
  • Filtered aggregates use one column as a mask and another as the values.

Verification Layer

  • Keep a known fast path as the baseline.
  • Compare new generic paths against known-correct output.
  • Check correctness and performance after each refactor.

4. Why Zig?

Analytical engines need systems programming and tight compute kernels at the same time. Zig is a good fit because it keeps memory explicit, exposes vector operations, and lets the programmer specialize code at compile time without adding a runtime abstraction layer.

Explicit Memory

Allocators, memory maps, and slices make ownership and data layout visible.

Direct SIMD

@Vector, @select, and @reduce describe vector kernels directly.

Compile-Time Code

comptime lets one function become specialized code for several numeric widths.

5. SIMD and Vectors

A scalar loop processes one value at a time. SIMD processes a small group of values at once. This is natural for column stores because values in one column are contiguous in memory. 

// Scalar idea: one value at a time.
for (values) |v| total += v;

// Vector idea: one chunk at a time.
const V = @Vector(lanes, i32);
const chunk: V = values[i..][0..lanes].*;
total += @reduce(.Add, chunk);

SIMD is not automatic magic. Stable speedups come from clear data layout, known numeric widths, chosen lane counts, and careful handling of the scalar tail.

6. How comptime Reduces Duplication

A database executor often runs the same algorithm over different integer widths. Instead of writing a separate function for every width, Zig can take the type as a compile-time argument. 

fn lanesFor(comptime T: type) comptime_int {
    return switch (T) {
        i16 => 32,
        i32 => 16,
        i64 => 8,
        else => @compileError("unsupported type"),
    };
}

fn loadVector(comptime T: type, values: []const T, i: usize) @Vector(lanesFor(T), T) {
    const lanes = lanesFor(T);
    return values[i..][0..lanes].*;
}

The call site stays simple, while the compiler still generates concrete SIMD code for each type. 

const a = loadVector(i16, small_values, 0); // 32 lanes
const b = loadVector(i64, big_values, 0);   // 8 lanes

This is not runtime reflection or a virtual call. The vector type and lane count are known at compile time.
