ZigHouse internals

从一条 SQL 到 SIMD 扫描

ZigHouse 是一个用 Zig 写的列式分析执行器。它的核心思想很朴素：把数据按列放好， 把 SQL 变成少量明确的计算计划，然后用 mmap、向量化和编译期泛型把 CPU 的吞吐打满。

1. ZigHouse 像什么？

如果传统行式数据库像一本账本，每一行是一条完整记录；列式数据库更像把账本拆成很多竖条： user_id 一列、event_date 一列、metric_value 一列。 分析查询通常只看少数列，所以列式执行可以少读很多数据。

列式

同一列连续存放，适合顺序扫描、SIMD reduce、mmap。

单机

先把单机 CPU、内存和文件系统吃透，再考虑更复杂的分布式形态。

分析

重点是 COUNT、SUM、AVG、MIN/MAX、过滤和聚合。

2. 最小端到端路径

通用路径不应该靠某个固定查询编号触发，而是从 SQL 文本解析出一个很小的计划。 例如： 

SELECT AVG(value_column)
FROM events
WHERE predicate_column <> 0;
SQL用户输入文本。
Parser识别 SELECT、聚合函数、表名和 WHERE。
Plan得到 aggregate plan。
Bind把列名绑定到 typed hot column。
Execute调用 SIMD kernel 输出 CSV。

3. 架构分层

查询层

  • 解析器把 SQL 文本转成一个很小的查询计划。
  • 绑定阶段把列名解析成具体的列数据。
  • 执行阶段选择适合列类型的计算内核。

存储层

  • hot columns 是固定宽度二进制列。
  • mmap 后直接把 bytes 视为 []const i16/i32/i64。
  • 扫描时不构造行对象，尽量避免分配。

执行层

  • COUNT(*) 直接用行数。
  • SUM/AVG/MIN/MAX 按列类型选择 SIMD kernel。
  • filtered aggregate 使用 predicate column 作为 mask。

性能保护层

  • 保留已知最快路径作为基线。
  • 新路径先和旧路径比较结果，再逐步扩大覆盖面。
  • 每次重构都要验证正确性和性能。

4. 为什么 Zig 适合做这个？

ZigHouse 需要同时处理系统编程和性能内核。Zig 的优势是：低层能力直接、编译期能力强、没有隐藏运行时。

显式内存

allocator、mmap、slice 都很直接，适合控制分配和生命周期。

直接 SIMD

@Vector、@select、@reduce 可以写出清晰的向量化 kernel。

comptime

用一个函数描述逻辑，让编译器为 i16/i32/i64 生成专门代码。

5. SIMD 和 vector 是什么？

普通循环一次处理一个数。SIMD 是一次处理一组数，比如 16 个 i32 或 32 个 i16。 对列式扫描来说，这非常自然，因为一列数据在内存里连续排列。 

// 标量思维：一次加一个
for (values) |v| total += v;

// SIMD 思维：一次读 lanes 个，向量相加，再 reduce
const V = @Vector(lanes, i32);
const v: V = values[i..][0..lanes].*;
total += @reduce(.Add, v);

SIMD 不等于“自动魔法”。要获得稳定收益，数据布局、类型宽度、lane 数、tail 处理都要明确。 一个好的执行器会把这些细节封装在少数通用计算内核里。

6. comptime 怎样减少重复代码？

数据库执行器经常要对不同整数宽度做同一种事情：扫描、过滤、求和、求最小值、求最大值。 如果每种类型都写一遍，代码会越来越像复制粘贴。Zig 的 comptime 可以把“类型”变成编译期参数， 用一个模板描述算法，再由编译器生成各个具体类型的机器码。 

fn lanesFor(comptime T: type) comptime_int {
    return switch (T) {
        i16 => 32,
        i32 => 16,
        i64 => 8,
        else => @compileError("unsupported type"),
    };
}

fn loadVector(comptime T: type, values: []const T, i: usize) @Vector(lanesFor(T), T) {
    const lanes = lanesFor(T);
    return values[i..][0..lanes].*;
}

这段代码表达的是“对一列做 reduce”。它不关心这一列具体是 16 位整数、32 位整数还是 64 位整数。 调用时类型是编译期已知的，向量宽度、累加器类型、初始值都会被编译器固定下来。 

const a = loadVector(i16, small_values, 0); // 32 lanes
const b = loadVector(i64, big_values, 0);   // 8 lanes

这不是运行时反射，也不是虚函数调用。每个具体类型仍然会生成专门的 SIMD 版本，只是源码不再为每个类型写一套函数。

本文档用 ZigHouse 作为背景，解释列式分析执行器的基本原理：列式布局、向量化扫描、编译期泛型和性能验证。
